
# 2. Подробная документация по проекту

## 2.1. Архитектура системы

Система управления роботом построена на базе ROS 2 Jazzy и состоит из 7 узлов, взаимодействующих через топики и UART-интерфейсы. Архитектура разделена на три функциональных слоя:

```
┌─────────────────────────────────────────────────────────────────┐
│                    СЛОЙ УПРАВЛЕНИЯ (ПДУ)                        │
│  ПДУ (геймпад PS5) ──LoRa 433──> gkv2_driver_node               │
│                                    │                            │
│                                    ▼ /cmd_vel                   │
├─────────────────────────────────────────────────────────────────┤
│                   СЛОЙ НАВИГАЦИИ (Jetson)                       │
│  ГКВ2 ──UART 921600──> gkv2_nav_parser_node                     │
│                              │ /nav/pose, /gkv2/status          │
│                              ▼                                  │
│                    navigation_fusion_node                       │
│                              │ /nav/fused_pose                  │
│                              ▼                                  │
│                    navigation_controller_node                   │
│                              │ /cmd_vel                         │
├─────────────────────────────────────────────────────────────────┤
│                    СЛОЙ ИСПОЛНЕНИЯ (БУД)                        │
│                    cmd_vel_bridge_node                          │
│                              │ UART 115200                      │
│                              ▼                                  │
│                         Моторы робота                           │
└─────────────────────────────────────────────────────────────────┘
```

## 2.2. Описание узлов

### 2.2.1. `gkv2_driver_node` — Драйвер ПДУ

**Назначение**: Приём команд от пульта дистанционного управления через LoRa 433 МГц и трансляция их в топик `/cmd_vel`.

**Вход**: UART `/dev/ttyV0` (921600 baud), пакет CMD_BUD (8 байт, CRC16).
**Выход**: Топик `/cmd_vel` (geometry_msgs/Twist).

**Логика**:
- Асинхронное чтение UART через Boost.Asio
- Парсинг пакета CMD_BUD согласно `Protokol_BUD.docx` (таблица 4.1)
- Преобразование процентов мощности (0-100) в линейную/угловую скорость
- Публикация команд с частотой 50 Гц

### 2.2.2. `gkv2_nav_parser_node` — Парсер навигации ГКВ2

**Назначение**: Приём и декодирование наборного пакета данных от блока ГКВ2.

**Вход**: UART `/dev/ttyV1` (921600 baud), пакет типа 0x13.
**Выход**: 
- `/nav/pose` (geometry_msgs/PoseStamped) — позиция и ориентация
- `/nav/imu` (sensor_msgs/Imu) — ускорения
- `/gkv2/status` (GKV2Status) — статусы ГНСС и алгоритма

**Ключевые особенности**:
- Парсинг по ID параметров из `params.yaml` (поле `packet_parameter_ids`)
- Корректное преобразование NED → ENU для ROS
- Разбор битовых полей `gps_state_status`, `gps_rel_status`, `alg_state_status`
- Статистика: счётчики пакетов и CRC-ошибок
- QoS: BEST_EFFORT для высокочастотных данных

**Парсинг статусов (согласно ЛМАП.402131.009Д1)**:
- `gps_state_status` (ID 72): биты 0,1 — годность даты/времени; бит 16 — годность координат; биты 22-23 — RTK статус (0=нет, 1=float, 2=fixed)
- `gps_rel_status` (ID 29): бит 8 — годность курса от двух антенн; биты 3-4 — статус неоднозначности
- `alg_state_status` (ID 96): биты 7-0 — этап алгоритма (50 = полная навигация); биты 31-16 — ошибки

### 2.2.3. `gkv2_config_node` — Узел конфигурации ГКВ2

**Назначение**: Одноразовая проверка конфигурации ГКВ2 при старте системы.

**Логика работы**:
1. Открывает UART `/dev/ttyV1` эксклюзивно
2. Отправляет пакет 0x00 (проверка соединения)
3. Отправляет пакет 0x04 (чтение данных об изделии: серийный номер, прошивка)
4. Отправляет пакет 0x26 (чтение текущей конфигурации наборного пакета)
5. Сравнивает полученный массив ID с ожидаемым из `params.yaml`
6. Публикует статус в `/gkv2/config_status` и завершается

**Запуск**: Через `OnProcessExit` в `main.launch.py` — остальные узлы стартуют только после завершения конфигуратора. Это исключает конфликт за порт `/dev/ttyV1`.

### 2.2.4. `odometer_node` — Эмулятор одометра

**Назначение**: Генерация сырой одометрии на основе команд `/cmd_vel` (до получения реального механического одометра).

**Вход**: `/cmd_vel` (geometry_msgs/Twist)
**Выход**: `/odom/raw` (nav_msgs/Odometry)

**Логика**: Интегрирует линейную и угловую скорости с учётом базы колёс (wheel_base = 0.7 м).

### 2.2.5. `navigation_fusion_node` — Слияние навигации

**Назначение**: Автоматическое переключение между ГНСС и одометрией при потере сигнала.

**Вход**: 
- `/nav/pose` от парсера ГКВ2
- `/gkv2/status` от парсера ГКВ2
- `/odom/raw` от одометра

**Выход**: 
- `/nav/fused_pose` (geometry_msgs/PoseStamped)
- `/nav/fused_odom` (nav_msgs/Odometry)
- `/nav/status` (NavigationStatus)
- tf2 transform `map → base_link`

**Алгоритм слияния**:
1. Рассчитывает качество ГНСС: `quality = 0.5*(RTK Fixed) + 0.3*(sats/20) + 0.2*(alg_navigation_ready)`
2. Если `quality >= 0.7` и `sats >= 8` и `rtk_status == 2` и таймаут < 2 сек → режим GNSS
3. Иначе → режим Odometry (интеграция cmd_vel)
4. При возврате к GNSS сбрасывает накопленный дрейф одометрии

**Расчёт качества ГНСС** (метод `calculate_gnss_quality`):
```cpp
double quality = 0.0;
if (status->rtk_status == 2) quality += 0.5;           // RTK Fixed
quality += 0.3 * std::min(status->gps_num_satellites / 20.0, 1.0);  // Спутники
if (status->alg_navigation_ready) quality += 0.2;      // Алгоритм готов
return quality;
```

### 2.2.6. `navigation_controller_node` — Навигационный контроллер

**Назначение**: Следование по контрольным точкам методом Pure Pursuit.

**Вход**: `/nav/fused_pose`, waypoints из `waypoints.yaml`
**Выход**: `/cmd_vel`

**Логика**: Pure Pursuit с настраиваемым lookahead distance. Загружает 5 waypoints из YAML при старте.

### 2.2.7. `cmd_vel_bridge_node` — Мост команд моторов

**Назначение**: Трансляция `/cmd_vel` в пакеты CMD_BUD для БУД через UART.

**Вход**: `/cmd_vel`, `/mode_switch` (std_msgs/Bool)
**Выход**: UART `/dev/ttyV4` (115200 baud), пакет CMD_BUD (8 байт)

**Режимы работы** (согласно п. 8.2 регламента «Дюна»):
- `MODE_OFF` (0) — ВЫКЛЮЧЕНО
- `MODE_PAUSE` (1) — ПАУЗА (нулевая мощность, IGN:ON)
- `MODE_DRIVE` (2) — ДВИЖЕНИЕ (передача cmd_vel, IGN:ON)
- `MODE_STOP` (3) — СТОП (нулевая мощность, IGN:OFF)

**Watchdog**: При отсутствии команд более 300 мс (`watchdog_timeout_sec`) — аварийная остановка.

**Частота отправки**: 50 Гц (20 мс), согласно п. 3.1.1 `Protokol_BUD.docx`.

**Формат пакета CMD_BUD** (таблица 4.1):
```
Байт 0: 0xDE (идентификатор)
Байт 1: Акселератор левый борт (0-100%)
Байт 2: Акселератор правый борт (0-100%)
Байт 3: Битовая маска флагов
        бит 0: направление левый (0=вперёд, 1=назад)
        бит 1: направление правый (0=вперёд, 1=назад)
        бит 2: зажигание (IGN)
        бит 3: тормоз 1
        бит 4: подрыв заряда
        бит 5-6: резерв
        бит 7: тормоз 2
Байты 4-5: CRC16 (алгоритм из Protokol_BUD.docx)
Байт 6: 0x0D
Байт 7: 0x0A
```

## 2.3. Протоколы связи

### 2.3.1. Входящий канал: ПДУ → Jetson (CMD_BUD)

- **Физический уровень**: LoRa 433 МГц → UART
- **Порт**: `/dev/ttyV0` (на Jetson: `/dev/ttyTHS0`)
- **Скорость**: 921600 baud
- **Частота**: 50 Гц
- **Формат**: 8 байт, CRC16 (таблица `cx_datagram_crc16_table` из `Protokol_BUD.docx`)
- **Гарантия доставки**: не гарантируется (п. 3.1.2)

### 2.3.2. Входящий канал: ГКВ2 → Jetson (навигация)

- **Физический уровень**: UART
- **Порт**: `/dev/ttyV1` (на Jetson: `/dev/ttyTHS1`)
- **Скорость**: 921600 baud
- **Частота**: 50-100 Гц (настраивается в ГКВ2)
- **Формат**: пакет типа 0x13, CRC32 (приложение А ЛМАП.402131.009Д1)
- **Структура**: `[0xFF][адрес][0x13][длина][данные...][CRC32 4 байта]`

### 2.3.3. Исходящий канал: Jetson → БУД (управление моторами)

- **Физический уровень**: UART
- **Порт**: `/dev/ttyV4` (на Jetson: `/dev/ttyTHS2`)
- **Скорость**: 115200 baud
- **Частота**: 50 Гц (20 мс)
- **Формат**: CMD_BUD, 8 байт, CRC16
- **Watchdog**: 300 мс (при отсутствии команд — аварийная остановка)

## 2.4. tf2 tree

```
map
 └── base_link  (публикуется navigation_fusion_node, 50 Гц)
```

Transform `map → base_link` содержит:
- `translation`: позиция робота в глобальной системе координат (ENU)
- `rotation`: ориентация в кватернионе (преобразование из Euler NED)
